gives you the flexibility to configure access based on your companyâ€™s specific operational and security needs
Users
Directly associated with an identity in an AWS account
can directly access AWS services with its own access keys and passwords
Usually created for real people or applications
users can be assigned specific permissions or certain policies can be applied to them
E.g.: create an AWS user for an application and give that user access only to the S3 bucket
User groups
for organizing and managing multiple AWS users
Ideal for grouping users with common permissions
A policy or permission assigned to a group is automatically assigned to all users in that group
Group them in a single "Developer" group => give same access permissions to all developers
Roles
Not directly associated with ID or password so cannot log in directly
used to transfer permissions to another AWS service or user for a certain period of time
frequently used to grant applications running on EC2 instances access to AWS services that those instances need
When an AWS service needs access to another service, a role is used to provide this access securely
EC2 instance writing data to S3 bucket => create a role for this EC2 instance + grant S3 write permissions
